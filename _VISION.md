## 1. Миссия

**"Ты не знаешь чего не знаешь."**

Разработчик не знает, что его `.env` читаем для всех процессов. Не знает, что `--no-verify` обходит pre-commit hook. Не знает, что Claude каждую сессию начинает с нуля без CLAUDE.md. Не знает, что его `NEXT_PUBLIC_` переменная утечёт в браузер.

Doctor находит эти слепые зоны, объясняет почему это проблема, и исправляет.

### Формулировка миссии (одно предложение)

> Doctor — автоматизированный аудит проекта для Claude Code: 42 проверки по 6 слоям, от секретов до DX. Находит пробелы, объясняет почему это важно, применяет проект-специфичные исправления.

### Метафора

Doctor — **ежегодный медосмотр для кодовой базы**. Ты можешь чувствовать себя здоровым, но рентген покажет то, что не видно снаружи. Не паникуй — просто знай и чини.

## 2. Проблема

### Корневая проблема

Claude Code из коробки — мощный инструмент, но требует настройки: hooks, skills, agents, rules, MCP, memory. Без этого:

| Область | Что происходит без Doctor | Последствия |
|---------|--------------------------|-------------|
| **Безопасность** | Секреты в git, `.env` 644, нет SAST | Утечка ключей, взлом, потеря денег |
| **Продуктивность** | Claude каждую сессию с нуля | Повторяешь одни и те же объяснения |
| **Качество** | Нет auto-lint при коммите | Сломанный код в main, баги в production |
| **DX** | Ручной ввод команд | Забываешь флаги, тратишь время |

### Gap Analysis — где именно пробел

```
Что ЕСТЬ у разработчика          Что НУЖНО            Кто закрывает gap
─────────────────────────────────────────────────────────────────────
ruff/eslint (линтер)              → Авто-lint при Edit  → Doctor (PostToolUse hook)
pytest/jest (тесты)               → Авто-тест при push  → Doctor (pre-push hook)
.env (секреты)                    → Защита от утечки    → Doctor (.gitignore + gitleaks)
Claude Code (AI)                  → Контекст проекта    → Doctor (CLAUDE.md + rules)
Git (версии)                      → Качество workflow   → Doctor (branch protection)
```

Doctor **не заменяет** инструменты — Doctor **связывает** их в работающую систему автоматизации.

### Почему существующие инструменты не решают

| Инструмент | Что делает | Чего НЕ делает | Покрытие vs Doctor |
|-----------|-----------|---------------|-------------------|
| **mega-linter** | 50+ линтеров | Не знает Claude Code ecosystem | 1 из 6 слоёв |
| **repo-linter** | Наличие LICENSE/README | Не проверяет содержимое | 0.5 из 6 слоёв |
| **Snyk/Dependabot** | CVE в зависимостях | Не .env, не hooks, не MCP | 0.5 из 6 слоёв |
| **GitHub scanning** | Secret patterns | Только GitHub, не локально | 0.3 из 6 слоёв |
| **pre-commit** | Git hooks | Один механизм, ручная настройка | 0.5 из 6 слоёв |
| **Doctor** | **42 проверки × 6 слоёв** | Не бизнес-логика, не рефакторинг | **6 из 6 слоёв** |

### Уникальность Doctor (5 столпов)

1. **Знает Claude Code** — hooks, skills, agents, rules, MCP, memory, plugins
2. **6 слоёв** — от chmod до smoke-тестов. Полная картина
3. **Объясняет ПОЧЕМУ** — не "исправь", а "исправь, потому что..." + источник
4. **Проект-специфичный** — читает ТВОЙ стек и генерирует ТВОИ конфиги
5. **Zero dependencies** — просто `.md` файлы. `curl | bash` и готово

## 3. Ценности продукта

### 3.1 Справедливость оценки

**Правило**: Severity = реальный импакт. "Что случится если НЕ исправить?"

| Вопрос | Если да | Severity |
|--------|---------|----------|
| Утечёт секрет? | Взлом, потеря денег | `red` |
| Сломается build? | Невозможно деплоить | `orange` |
| Ухудшится DX? | Потеря времени | `yellow` |
| Просто некрасиво? | Ничего не случится | `blue` |

**Примеры правильной оценки:**
- Personal проект без CI → `blue` (не `red`)
- `print()` в CLI → OK (не проблема)
- `.env` 644 на Mac с одним юзером → `blue` (не `orange`)
- Нет MCP/agents если Claude Code не используется → N/A (не `orange`)

### 3.2 Конкретность

**Правило**: Каждая находка = одна конкретная команда или один файл.

```
Плохо:  "Улучшите безопасность проекта"
Хорошо: "chmod 600 .env .mcp.json"

Плохо:  "Добавьте тесты"
Хорошо: "Создайте tests/test_main.py с pytest, покрывающий calculate_total()"

Плохо:  "Настройте CI"
Хорошо: "Создайте .github/workflows/ci.yml: lint → test → SAST"
```

### 3.3 Security First

**Правило**: Слой 0 проверяется ПЕРВЫМ. Секретная `red` > все `orange` остальных слоёв.

Порядок приоритета при лечении:
1. Утёкшие секреты → ротировать НЕМЕДЛЕННО
2. Секреты без защиты → .gitignore + chmod + gitleaks
3. Отсутствие auto-check → PostToolUse + pre-commit
4. Всё остальное

### 3.4 Проект-специфичность

**Правило**: Generic шаблон = провал.

```
Плохо:  PostToolUse → "eslint" (а проект на Python)
Хорошо: PostToolUse → "ruff check" (увидел ruff в pyproject.toml)

Плохо:  "Добавь pytest" (а проект на Rust)
Хорошо: "cargo test" (увидел Cargo.toml)
```

Алгоритм: **Прочитай стек → адаптируй рекомендацию → проверь что работает**

### 3.5 Не навреди (Primum non nocere)

**Правило**: Лучше пропустить, чем сломать. Лучше спросить, чем откатывать.

- НИКОГДА не перезаписывать работающий конфиг без спроса
- НИКОГДА не коммитить без одобрения
- НИКОГДА не хранить секреты в коммитируемых файлах
- При сомнении → спроси пользователя

### 3.6 Объясняй

**Правило**: Если не можешь объяснить ПОЧЕМУ за 2 предложения — это не реальная проблема.

Формат находки:
```
Команда/действие
  Почему это проблема (1-2 предложения)
  -> источник (URL)
```

## 4. Позиционирование

### Doctor — ЭТО

- Автоматизированный аудит автоматизации проекта
- 42 проверки × 6 слоёв
- Диагностика → рецепт → лечение (медицинская метафора)
- 20+ стеков из коробки
- Каждая находка с ПОЧЕМУ + источник
- Проект-специфичные фиксы (не шаблоны)
- Skill для Claude Code (`.md`, zero deps)
- Open source, MIT

### Doctor — НЕ ЭТО

- НЕ линтер (есть ruff/eslint/clippy)
- НЕ сканер CVE (есть Snyk/Dependabot)
- НЕ тест-фреймворк (есть pytest/jest)
- НЕ CI/CD (есть GitHub Actions)
- НЕ рефакторинг (не бизнес-логика)
- НЕ boilerplate generator
- НЕ замена code review

### Elevator Pitch (15 секунд)

> Doctor за 2 минуты проверит 42 точки твоего проекта — от утечки секретов до отсутствия smoke-тестов. Объяснит каждую проблему и исправит под твой стек. Не шаблонами, а конкретно.

## 5. Архитектура

### Файловая структура

```
claude-doctor-skill/
  SKILL.md              — Мозг: протокол, подкоманды, формат, guardrails (~240 строк)
  CHECKLIST.md          — Индекс: 42 проверки × 6 слоёв, краткие, со ссылками
  README.md             — Витрина: install, usage, comparison
  LICENSE               — MIT
  install.sh            — curl | bash installer
  layers/
    SECURITY.md         — Слой 0: 11 проверок + incident response plan (573 строк)
    FOUNDATION.md       — Слой 1: 5 проверок (507 строк)
    QUALITY.md          — Слой 2: 7 проверок (544 строк)
    QUALITY-EXTRA.md    — Слой 2: 4 проверки (343 строк)
    INTELLIGENCE.md     — Слой 3: 2 проверки (231 строк)
    CONTEXT.md          — Слой 4: 5 проверок (445 строк)
    DX.md               — Слой 5: 7 проверок (564 строк)
  assets/
    demo-scan.gif       — Демо /doctor scan
    demo-fix.gif        — Демо /doctor fix
    demo.svg            — Бейдж для README
  _VISION.md            — [gitignored] Этот файл
  _PERSONAS.md          — [gitignored] Целевая аудитория
```

### Принципы архитектуры

| Принцип | Правило | Почему |
|---------|---------|--------|
| Компактный мозг | SKILL.md < 250 строк | Claude теряет фокус на длинных промптах |
| Индекс + детали | CHECKLIST = краткий, layers = подробно | Двухуровневая навигация |
| Самодостаточность | Каждый layer-файл читается отдельно | `/doctor layer 3` не нужны другие слои |
| Zero deps | Только `.md` файлы | `curl | bash` без npm/pip/docker |
| Modular boundary | Файл > 400 строк → разбей | QUALITY → QUALITY + QUALITY-EXTRA |
| Context budget | Layer файлы загружаются по запросу | Не тратим контекст на ненужные слои |

## 7. Формула приоритизации

```
Priority = (Impact × Reach × Frequency) / Effort
```

| Параметр | 1 | 3 | 5 |
|----------|---|---|---|
| **Impact** | Косметика | Ухудшает DX | Утечка секретов |
| **Reach** | Экзотический стек | 50% проектов | Все проекты |
| **Frequency** | Раз в год | Раз в месяц | Каждый проект |
| **Effort** | Переписать SKILL.md | Новый layer-файл | Добавить строку |

### Примеры расчёта

| Фича/проверка | I | R | F | E | Score | Решение |
|---------------|---|---|---|---|-------|---------|
| .env permissions check | 4 | 5 | 5 | 1 | **100** | Есть |
| SAST проверка | 5 | 4 | 3 | 2 | **30** | Есть |
| `/doctor quick` TOP-3 | 3 | 5 | 5 | 2 | **37.5** | v2.1 |
| Effort estimation | 2 | 5 | 5 | 2 | **25** | v2.1 |
| Kubernetes manifests | 3 | 2 | 2 | 4 | **3** | Нет |
| Terraform state | 4 | 1 | 1 | 3 | **1.3** | Нет |

### Пороги

- **Score >= 15** → добавляй в текущую версию
- **Score 5-15** → обсуди, может быть в следующей
- **Score < 5** → не добавляй, YAGNI

### Когда УДАЛЯТЬ проверку

- Технология мертва (устарела)
- Дублирует другую проверку
- False positive rate > 30%
- Effort поддержки > Impact

## 9. Тон и голос

### Принцип

Doctor говорит как **опытный коллега** — не менторствует, не пугает, не занудствует. Уважает время.

### Правильно vs Неправильно

| | Правильно | Неправильно |
|---|-----------|-------------|
| **Находка** | `chmod 600 .env` + почему + ссылка | "ВНИМАНИЕ! КРИТИЧЕСКАЯ УЯЗВИМОСТЬ!" |
| **Результат** | `Тесты: 45 pass, 2 fail — src/utils.py:42` | "При запуске были обнаружены некоторые проблемы..." |
| **Оценка** | `Безопасность: 8/11 (73%) — ХОРОШО` | "В целом безопасность находится на удовлетворительном уровне" |
| **Действие** | `Исправить все 3? Это ~5 минут.` | "Рекомендуется рассмотреть возможность..." |

### Шкала серьёзности

| Эмодзи | Уровень | Тон | Пример |
|--------|---------|-----|--------|
| `red` | Критично | Факт, без паники | "Секрет в git истории. Ротируй ключ." |
| `orange` | Важно | Рекомендация | "Добавь gitleaks в CI — pre-commit обходится." |
| `yellow` | Средне | Совет | "Lock-файл устарел на 3+ месяца." |
| `blue` | Мелочь | Заметка | "Можно добавить smoke-тест." |

### Адаптация под персону (автоматическая)

Doctor **не спрашивает** уровень пользователя. Вместо этого:
- Проект без `.claude/` → вероятно новичок → подробнее объясняй
- Проект с agents + rules + hooks → опытный → кратко
- OSS проект (LICENSE + CONTRIBUTING) → мейнтейнер → фокус на DX

### Язык

- Определяется автоматически по языку юзера
- Технические термины НЕ переводятся: hook, skill, agent, MCP, pre-commit, CI
- Русский для русскоязычных, English для англоязычных

## 11. Антипаттерны

| Антипаттерн | Что это | Как избежать |
|-------------|---------|-------------|
| **Feature creep** | Добавляем "потому что можем" | Формула приоритизации (Score >= 15) |
| **Severity inflation** | `blue` → `red` | Задай вопрос: "Что случится если НЕ исправить?" |
| **Generic templates** | `pre-commit-config.yaml` без изучения проекта | Фаза 1 обязательна перед любыми изменениями |
| **Silent fixes** | Меняем файлы без спроса | Фаза 4: "Исправить все? Или по одной?" |
| **Scope creep** | Лезем в бизнес-логику | Границы: автоматизация, НЕ рефакторинг |
| **Vendor lock-in** | Привязка к GitHub/Railway | Рекомендации под стек, не под платформу |
| **Complexity creep** | SKILL.md > 250 строк | Modular boundary: выноси в layers/ |
| **Paternalism** | Решаем за юзера | Спрашиваем перед каждым изменением |

## 12. Принципы решений (в порядке приоритета)

1. **Не навреди** — фича может сломать проект? Не делай
2. **Security > DX** — безопасность важнее удобства
3. **Проект-специфичность > универсальность** — лучше отлично для Python, чем средне для всех
4. **Объясни > исправь** — лучше объяснить, чем молча исправить
5. **YAGNI** — не добавляй то, что не нужно СЕГОДНЯ
6. **Простота > мощность** — простой формат > мощный но сложный
7. **Факты > мнения** — каждая рекомендация обоснована

### Decision Tree для спорных случаев

```
Добавить новую проверку?
  ├── Score >= 15? → ДА
  ├── Score 5-15?  → ОБСУДИ (добавь в roadmap)
  └── Score < 5?   → НЕТ (YAGNI)

Изменить severity?
  ├── Утечёт секрет?        → red
  ├── Сломается build?       → orange
  ├── Ухудшится DX?          → yellow
  └── Просто некрасиво?      → blue

Добавить в SKILL.md или в layer?
  ├── Нужно при КАЖДОМ запуске? → SKILL.md
  └── Нужно при запросе слоя?   → layers/*.md

Написать на русском или английском?
  └── На языке юзера (автоопределение)
```

## 13. Известные слабости и как их закрыть

Честный взгляд на текущие ограничения Doctor:

| Слабость | Суть | Как закрыть | Версия |
|----------|------|-------------|--------|
| **Нет persistence** | Результат аудита не сохраняется | Checkpoint файл `.claude/doctor-checkpoint.json` | v2.1 |
| **Нет diff** | Повторный запуск = полный аудит с нуля | `/doctor diff` с checkpoint | v2.1 |
| **Information overload** | 42 проверки разом — пугает новичков | `/doctor quick` TOP-3 | v2.1 |
| **Нет терминологии** | Новички не знают "hook", "MCP" | Glossary в SKILL.md | v2.1 |
| **Нет CI интеграции** | Doctor = только локально | GitHub Action | v3.0 |
| **Нет кастомизации** | Нельзя добавить свои проверки | `.claude/doctor-rules/` | v3.0 |
| **Нет batch** | 5 проектов = 5 ручных запусков | Batch mode | v3.0 |
| **SKILL.md ~240 строк** | Близко к лимиту 250 | Рефакторинг: вынести формат в отдельный файл | v2.1 |
| **Layer файлы 400-570 строк** | Некоторые выше modular boundary | Разбить SECURITY.md и DX.md | v2.1 |

## 14. User Journey Map

### Путь нового пользователя

```
1. ОБНАРУЖЕНИЕ
   "Увидел в Telegram/GitHub/Twitter"
   └── README → GIF демо → "О, прикольно"

2. УСТАНОВКА (~30 секунд)
   curl -sSL ... | bash
   └── 9 файлов, zero deps, готово

3. ПЕРВЫЙ ЗАПУСК (~2 минуты)
   /doctor scan
   └── Профиль проекта → 6 слоёв → оценка X/42 (N%)
   └── "Вау, не знал про chmod 600"

4. ПЕРВЫЙ FIX (~5 минут)
   /doctor fix
   └── "Исправить все 7? Или по одной?"
   └── Автоматические фиксы под стек

5. ВЕРИФИКАЦИЯ
   /doctor verify
   └── "ЗДОРОВЬЕ: 38/42 (90%) — ОТЛИЧНО"

6. ПРИВЫЧКА
   /doctor (периодически)
   └── "Что изменилось?" → /doctor diff (v2.1)

7. РЕКОМЕНДАЦИЯ
   "Попробуй Doctor, мне помогло"
   └── Star + share
```
